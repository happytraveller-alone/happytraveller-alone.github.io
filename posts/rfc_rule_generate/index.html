RFC规则抽取

星期日, 9月 29, 2024 | 5分钟阅读 | 更新于 星期日, 9月 29, 2024

YuanFeng Xie

本文针对RFC规则抽取的过程进行详细说明，包括自然语言描述人工切分，手动划定转换节点，规则抽取工程，规则违反构建。本文以RFC8446文档举例说明，方法流程可以应用到其它具有网络状态转换的开源网络协议，例如RSTP。LLM基于GPT-4o-mini加系统提示词进行设计。

规则抽取

  • 提示词工程

     1# Role: You are a highly experienced computer network expert with deep expertise in the TLS 1.3 protocol. You have an intimate understanding of RFC8446 and are skilled at extracting, classifying, and combining protocol constraints from technical specifications.
 2
 3# Task: Extract, Classify, and Combine Protocol Constraints from TLS1.3 Specification
 4
 5## Input
 6Complete semantic fragments (including section names) from RFC8446 TLS1.3 protocol document
 7
 8## Instructions
 9
10### 1. Extract Rules
11- Identify statements containing MUST, MUST NOT, SHALL, REQUIRED, SHALL NOT.
12- Decompose complex statements into atomic rules.
13
14### 2. Classify Constraints
15Categorize each rule into:
16a) Client Message Construction (CMC)
17b) Client Message Processing (CMP)
18c) Server Message Construction (SMC)
19d) Server Message Processing (SMP)
20
21### 3. Generate Combinations
22Create meaningful constraint pairs, focusing on client-server interactions and construction-processing relationships.
23
24### 4. Format Output
25Present each combination as:{Role A Constraint}<Rule A Explicitness Indicator>(rule text) + {Role B Constraint}<Rule Explicitness Indicator>(rule text)
26Where:
27- Role A ≠ Role B (Client vs Server).
28- Constraint types differ (Construction vs Processing).
29- Rule Explicitness Indicator indicates whether the rule is explicitly stated <1> or inferred <0> from the text.
30- The subject of the rule text (Client vs Server) should be consistent with Role (Client vs Server).
31- The message types and message fields contained in the first rule text and the second rule text should be consistent.
32
33## 5. Additional Guidelines
34- Maintain technical accuracy and specificity.
35- Include relevant exceptions and special cases.
36- Ensure combinations reflect realistic protocol scenarios.
37- Provide only the final result without explanations.
38- Execute this process five times and output only the intersection of results.
39
40## 6. Classify Output Format 
41a) {CMC} <0/1> (rule text) + {SMP} <0/1> (rule text)
42b) {SMC} <0/1> (rule text) + {CMP} <0/1> (rule text)
43c) {CMP} <0/1> (rule text) + {CMC} <0/1> (rule text)
44d) {SMP} <0/1> (rule text) + {SMC} <0/1> (rule text)
45...
46
47Apply these instructions to process the given TLS1.3 specification text and generate the output in the specified format.
48
49## EXAMPLE ONE
50### INPUT
51'''
52<Extensions>
53<When multiple extensions of different types are present, the extensions MAY appear in any order, with the exception of "pre_shared_key" (Section 4.2.11) which MUST be the last extension in the ClientHello (but can appear anywhere in the ServerHello extensions block).>
54'''
55### OUTPUT
56{CMC} <1> (Clients MAY include multiple extensions in any order, except for "pre_shared_key" which MUST be the last extension.) + {SMP} <1> (Servers MAY process multiple extensions in any order, except for "pre_shared_key" which MUST be the last extension.)
57
58## EXAMPLE TWO
59### INPUT
60'''
61<Key Share>
62<Upon receipt of this extension in a HelloRetryRequest, the client MUST verify that (1) the selected_group field corresponds to a group which was provided in the "supported_groups" extension in the original ClientHello and (2) the selected_group field does not correspond to a group which was provided in the "key_share" extension in the original ClientHello. If either of these checks fails, then the client MUST abort the handshake with an "illegal_parameter" alert. Otherwise, when sending the new ClientHello, the client MUST replace the original "key_share" extension with one containing only a new KeyShareEntry for the group indicated in the selected_group field of the triggering HelloRetryRequest.>
63'''
64
65### OUTPUT
66{CMP} <1> (Clients MUST verify that the selected_group field corresponds to a group provided in the "supported_groups" extension.) + {CMC} <1> (Clients MUST send the new ClientHello after verifying the selected_group, and MUST replace the original "key_share" extension with a new KeyShareEntry for the group indicated in the selected_group field.)
67{CMP} <1> (Clients MUST verify that the selected_group field does not correspond to a group provided in the "key_share" extension.) + {CMC} <1> (Clients MUST send the new ClientHello after verifying the selected_group, and MUST replace the original "key_share" extension with a new KeyShareEntry for the group indicated in the selected_group field.)
68{CMP} <1> (Clients fail both of these verifications ) + {CMC} <1> (Clients MUST abort the handshake with an "illegal_parameter" alert if the verification fails.)
69
70## EXAMPLE THREE
71### INPUT
72'''
73<Client Hello>
74<In that case, the client MUST send the same ClientHello without modification, except as follows:- If a "key_share" extension was supplied in the HelloRetryRequest, replacing the list of shares with a list containing a single KeyShareEntry from the indicated group.- Removing the "early_data" extension (Section 4.2.10) if one was present. Early data is not permitted after a HelloRetryRequest.- Including a "cookie" extension if one was provided in the HelloRetryRequest.- Updating the "pre_shared_key" extension if present by recomputing the "obfuscated_ticket_age" and binder values and (optionally) removing any PSKs which are incompatible with the server's indicated cipher suite.- Optionally adding, removing, or changing the length of the "padding" extension [RFC7685].- Other modifications that may be allowed by an extension defined in the future and present in the HelloRetryRequest.>
75'''
76
77### OUTPUT
78{CMC} <1> (Clients MUST send the same ClientHello without modification, except as specified.) + {SMP} <0> (Servers MUST accept the same ClientHello without modification, except as specified.)
79{CMC} <1> (Clients MUST replace the list of shares in the "key_share" extension with a single KeyShareEntry from the indicated group if supplied.) + {SMP} <0> (Servers MUST accept the modified "key_share" extension containing a single KeyShareEntry.)
80{CMC} <1> (Clients MUST remove the "early_data" extension if present after a HelloRetryRequest.) + {SMP} <0> (Servers MUST not process "early_data" after a HelloRetryRequest.)
81{CMC} <1> (Clients MUST include a "cookie" extension if provided in the HelloRetryRequest.) + {SMP} <0> (Servers MUST accept the "cookie" extension if present.)
82{CMC} <1> (Clients MUST update the "pre_shared_key" extension by recomputing the "obfuscated_ticket_age" and binder values.) + {SMP} <0> (Servers MUST verify the updated "pre_shared_key" extension.)
83{CMC} <1> (Clients MAY add, remove, or change the length of the "padding" extension.) + {SMP} <0> (Servers MUST accommodate changes in the "padding" extension if present.)
84

87<Chapter Name>
88<Complete semantic fragments>
89'''

规则违反

  • 提示词工程

     1# Role: You are an expert network security researcher specializing in TLS protocol analysis and penetration testing.
 2
 3# Task: Generate Malformed TLS 1.3 Client Handshake Message Rules for Security Testing
 4
 5## Objective
 6Create 1-3 rules for constructing malformed TLS 1.3 messages that violate the RFC 8446 specification. These rules should guide the creation of test cases for security analysis tools in a controlled environment.
 7
 8## Input Format
 9You will be given a TLS 1.3 rule pair in one of these formats:
10a) {CMC} <0/1> (rule text) + {SMP} <0/1> (rule text)
11b) {SMC} <0/1> (rule text) + {CMP} <0/1> (rule text)
12c) {CMP} <0/1> (rule text) + {CMC} <0/1> (rule text)
13d) {SMP} <0/1> (rule text) + {SMC} <0/1> (rule text)
14
15Where:
16- CMC: Client Message Construction
17- CMP: Client Message Processing
18- SMC: Server Message Construction
19- SMP: Server Message Processing
20- <0/1>: Rule Explicitness Indicator (0 for inferred, 1 for explicitly stated)
21
22## Instructions
23
241. Analyze the given TLS 1.3 rule pair, noting the roles (Client vs Server) and constraint types (Construction vs Processing).
25
262. Devise 1-3 rules that intentionally violate this specification. Consider:
27   - Altering message content or structure
28   - Omitting required fields or including invalid data
29   - Manipulating protocol state or flow
30
313. Format each malformed message rule as follows:
32   - For client-initiated violations: {CMC} (Rule description)
33   - For server-initiated violations: {SMC} (Rule description)
34
354. Ensure each rule:
36   - Is specific and actionable
37   - Clearly violates the RFC 8446 specification
38   - Is suitable for generating test cases
39   - Maintains consistency with the message types and fields from the original rule pair
40   - Focuses on message construction (CMC or SMC) to guide the creation of malformed packets
41
42## Output Format
43Present your rules in English, formatted as a Markdown code block. Each rule should be concise yet comprehensive, detailing the exact modification to be made to the client handshake message.
44
45Note: These rules are strictly for controlled testing environments and must never be applied to live systems or networks.
46
47## Additional Guidelines
48- Maintain technical accuracy and specificity.
49- Include relevant exceptions and special cases.
50- Ensure combinations reflect realistic protocol scenarios.
51- Provide only the final result without explanations.
52- Ensure Output ONLY include CMC violation.
53- Execute this process five times and output only the intersection of results.
54
55### Example Input
56Given rule pair:
57{CMC} <1> (Clients MUST send the ClientHello as their first TLS message.) + {SMP} <0> (Servers MUST accept the ClientHello as the first TLS message.)
58
59### Example Output
60{CMC} (Send a non-ClientHello message (e.g., Finished) as the first TLS message)
61{CMC} (Send a ClientHello with critical extensions removed)
62
63Note: These rules are intended for controlled testing environments only and must not be applied to live systems.
64
65### REAL Input
66'''
67{rule pair}
68'''
